## [CachyLLama: llama.cpp Fork with Persistent KV Cache for Long Agent Sessions](https://www.reddit.com/r/LocalLLaMA/comments/1v5k08a/cachyllamas_llamacpp_fork_with_persistent_kv/) ⭐️ 8.0/10

CachyLLama, a fork of the widely used llama.cpp local LLM inference library, introduces persistent SSD-backed KV cache checkpoints and multi-tier caching to eliminate redundant repeated prompt processing. This change drastically reduces unnecessary computation for long local agent sessions on consumer hardware. This fork addresses a high-impact, widespread pain point for local LLM users, particularly those running agentic workflows on lower-end consumer hardware. By eliminating redundant prompt processing, it makes long local agent sessions substantially more responsive and usable for a broad range of users. The persistent KV cache checkpoints survive server restarts, and the fork includes specialized handling for hybrid architectures including Qwen 3.5/3.6, Gemma 4, and GLM-4.7 where recurrent state restoration is more complex than standard attention-only KV caches. Project benchmarks run on a 7840U/780M test system report warm prompt processing times of 0.41s for 1,243-token prompts, 0.57s for 5,409-token prompts, and 0.99s for 15,700-token prompts, compared to cold times of 9.3s, 43.3s, and 143.1s respectively.

reddit · r/LocalLLaMA · /u/UsualResult · Jul 24, 18:39

**Background**: llama.cpp is the de facto standard open-source library for local large language model inference, serving as the core of popular tools including Ollama and LM Studio. KV \(key-value\) caching is a common transformer inference optimization that stores intermediate computation results of processed prompt tokens to avoid re-running identical calculations for repeated requests. Standard llama.cpp only stores KV cache in volatile GPU or system memory, which is lost on server restart, and does not support persistent caching for unchanged prompt segments, leading to redundant computation for long agent sessions that resend large system prompts and full conversation history on every request.

## [Postgres LISTEN/NOTIFY Scales to 60,000 Events Per Second](https://www.dbos.dev/blog/postgres-listen-notify-scalability) ⭐️ 7.0/10

A new technical analysis from the DBOS team demonstrates that PostgreSQL&\#x27;s LISTEN/NOTIFY feature can scale to 60,000 events per second for appropriate use cases, directly countering prior widespread claims that the feature has inherent, severe performance limitations. The post details optimizations that enable this level of throughput on a single Postgres server with millisecond-scale latency. This clarification is significant for backend engineers building event-driven or real-time applications, as it expands the set of viable use cases for a built-in Postgres feature that eliminates the operational overhead of dedicated message brokers like Redis or Kafka. It also resolves a long-running debate in the PostgreSQL community about the feature&\#x27;s production readiness for moderate-to-high throughput workloads.
